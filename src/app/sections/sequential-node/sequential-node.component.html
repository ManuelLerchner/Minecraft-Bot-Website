Sequential Node:

The next thing you may want to do with different TaskNodes / Actions is to chain them together, so that the bot can execute multiple tasks in a sequential order.
This is exactly what the SequentialNode is doing. It receives an arbitrary amount of children-nodes and its task is to compile each child and then connect it with its successor.
This act of linking the children isn't limited to ActionNodes only:
We have previously defined the compile() function in a way, that every compilation returns an entry- and an exit-state for the compiled node. So by using the entry and exit-states, we can also use other SequentialNodes inside the current SequentialNode, and it isn't really constrained to just sequential-nodes, we can use all possible ASTNodes as children.
Each child will then get recursively compiled and connected with its successor along the way. This allows for arbitrary nesting of different nodes, and thus creating a real hierarchical-program.

As mentioned above, when you compile a SequentialNode it recursively compiles all its children and connects them with newly created transitions.
Additionally it creates an entry- and an exit-state (called "Sequential-Node" and "End Sequential-Node") for the node. These special states have no impact on the minecraft-bot. Their whole purpose is to allow us easily verify if the generated-program is correct, because those states allow us to match each action to its corresponding parent-node. Furthermore, they also provide a simple way of entering and exiting a node.
